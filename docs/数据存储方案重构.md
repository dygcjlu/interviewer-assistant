# 数据存储方案重构

## 背景与目标

当前项目同时维护 SQLite 数据库和本地文件（简历 PDF/MD、录音），存在数据冗余（`resume_text` 同时在 DB 和 `.md` 文件中）。重构目标：

1. **Agent 选择性读取**：Agent 通过文件层级导航（先读轻量 index，再按需钻取具体文件），而非全量加载
2. **人类可直接查看**：所有候选人数据以 Markdown 存储，可在任意编辑器中浏览
3. **简化基础设施**：彻底移除 SQLite 和 `aiosqlite` 依赖

现有 SQLite 数据可直接丢弃，全新开始。

---

## 新目录结构

```
candidates/
├── index.md                          # 全局候选人目录（agent 导航入口）
└── {candidate_id}/                   # 例：c-abc123def456
    ├── profile.md                    # 候选人档案（YAML frontmatter + 简历全文）
    ├── resume.pdf                    # 原始 PDF（从 resumes/ 迁移命名规则）
    └── interviews/
        ├── index.md                  # 本候选人的面试历史摘要（agent 导航）
        └── {interview_id}/           # 例：i-789xyz
            ├── questions.md          # 面试问题清单
            ├── transcript.md         # 完整对话记录（面试结束后写入）
            ├── eval_report.md        # 评价报告（可选，生成后写入）
            └── session.json          # 会话元数据（轻量 JSON）
```

> `resumes/` 目录：本次重构为一次性切换，旧 SQLite 数据连同 `resumes/` 下已有文件直接丢弃，不做迁移。新上传简历统一写入 `candidates/{id}/resume.pdf`；`resumes/` 目录可在重构完成后手动删除。

## 各文件格式

### candidates/index.md

全局候选人目录，Agent 读取此文件后能决定下一步需要读哪个候选人的 profile。

文件同时包含两部分：YAML frontmatter 存结构化列表（供 REST API 解析返回 JSON），正文 Markdown 表格供人类阅读。两部分由 `MemoryModule` 写入时同步维护，内容完全一致。

```markdown
---
candidates:
  - id: c-abc123
    name: 张三
    created_at: "2025-01-15"
    latest_interview: "2025-03-10"
  - id: c-def456
    name: 李四
    created_at: "2025-02-01"
    latest_interview: null
---

# 候选人目录

| 候选人 | ID | 创建时间 | 最近面试 |
|---|---|---|---|
| 张三 | c-abc123 | 2025-01-15 | 2025-03-10 |
| 李四 | c-def456 | 2025-02-01 | — |
```

每次新增或删除候选人时同步更新此文件（frontmatter 和正文表格均更新）。`search_candidates(keyword)` 在读取 frontmatter 后做 in-memory 姓名过滤；`get_candidate_by_name(name)` 做精确匹配，用于上传去重。

### candidates/{id}/interviews/index.md

本候选人的面试历史摘要，Agent 读此文件后能决定是否需要深入读某次具体面试。

与 `candidates/index.md` 相同，文件包含两部分：YAML frontmatter 存结构化面试列表（供 `MemoryModule` 解析），正文 Markdown 表格供人类阅读。

```markdown
---
interviews:
  - interview_id: i-789xyz
    start_time: "2025-03-10T14:00:00+08:00"
    end_time: "2025-03-10T15:30:00+08:00"
    stage: completed
    trigger_mode: auto
    overall_score: 8.5
    recommendation: 推荐录用
    key_findings: 技术扎实，系统设计突出，沟通清晰
  - interview_id: i-456abc
    start_time: "2025-01-20T10:00:00+08:00"
    end_time: "2025-01-20T11:00:00+08:00"
    stage: completed
    trigger_mode: manual
    overall_score: 6.0
    recommendation: 待定
    key_findings: 基础尚可，项目经验偏浅，需进一步观察
---

# 张三 · 面试历史

| 面试 ID | 开始时间 | 状态 | 触发模式 | 综合评分 | 推荐结论 | 关键结论 |
|---|---|---|---|---|---|---|
| i-789xyz | 2025-03-10 14:00 | 已完成 | 自动 | 8.5/10 | 推荐录用 | 技术扎实，系统设计突出，沟通清晰 |
| i-456abc | 2025-01-20 10:00 | 已完成 | 手动 | 6.0/10 | 待定 | 基础尚可，项目经验偏浅，需进一步观察 |
```

`start_time`/`end_time`/`stage`/`trigger_mode` 字段供 `GET /api/candidates/{id}/history` 接口直接组装 `InterviewSummary` 返回，无需额外读取 `session.json`。`key_findings` 字段（1~2 句话）是 Agent 的第二层导航依据：仅凭评分无法判断是否需要深读历史，而关键结论能让 Agent 快速决定是否有必要读完整 `eval_report.md` 或 `transcript.md`。评价报告生成后由 `MemoryModule` 同步更新 frontmatter 和正文表格中的对应字段。

### candidates/{id}/interviews/{id}/session.json

轻量会话元数据，保存不适合 Markdown 表达的结构化字段。

```json
{
  "interview_id": "i-789xyz",
  "candidate_id": "c-abc123",
  "start_time": "2025-03-10T14:00:00+08:00",
  "end_time": "2025-03-10T15:30:00+08:00",
  "stage": "completed",
  "trigger_mode": "auto",
  "recording_candidate_path": "recordings/i-789xyz/candidate.wav",
  "recording_interviewer_path": "recordings/i-789xyz/interviewer.wav",
  "context_summary": "前5轮已讨论分布式系统设计和消息队列，候选人对Kafka理解扎实..."
}
```

`context_summary` 字段存储 `ContextManager` 对历史对话的压缩摘要（对应 `InterviewSession.context_summary`），面试结束时写入，为后续评价和历史回顾提供参考。面试进行中该字段为空字符串或不存在。

### candidates/{id}/interviews/{id}/questions.md

面试问题清单，面试准备阶段生成，面试结束后记录覆盖状态。

文件包含两部分：YAML frontmatter 存结构化问题列表（供 `MemoryModule` 解析反序列化为 `list[InterviewQuestion]`），正文 Markdown 供人类阅读和 Agent 直接消费。

```markdown
---
questions:
  - id: 1
    dimension: 系统设计
    difficulty: medium
    question: 请设计一个支持百万并发的消息队列系统。
    follow_ups:
      - 如何保证消息不丢失？
      - 如何处理消费者积压？
    is_covered: true
  - id: 2
    dimension: 项目经验
    difficulty: medium
    question: 介绍你在字节跳动最有挑战性的项目。
    follow_ups: []
    is_covered: false
---

# 面试问题清单

## 系统设计（中等难度）

- [x] 请设计一个支持百万并发的消息队列系统。
  - 追问：如何保证消息不丢失？
  - 追问：如何处理消费者积压？

## 项目经验（中等难度）

- [ ] 介绍你在字节跳动最有挑战性的项目。
```

`MemoryModule` 读取时解析 frontmatter 反序列化；写入时 frontmatter 和正文同步生成。`get_latest_question_plan(candidate_id)` 的新实现：读 `candidates/{id}/interviews/index.md` 的 frontmatter 取最新 `interview_id`，再读对应 `questions.md` 的 frontmatter 返回结构化列表。

## Agent 导航模式

Agent 按以下层级选择性读取，避免全量加载：

```
第一层（必读）
  candidates/index.md
    ↓ 根据候选人名字/ID 定位
第二层（按需）
  candidates/{id}/profile.md                    ← 需要了解候选人背景时
  candidates/{id}/interviews/index.md           ← 需要了解历史面试时
    ↓ 根据 关键结论/日期/评分 决定是否深入
第三层（按需）
  candidates/{id}/interviews/{iid}/questions.md   ← 准备面试/查看题目时
  candidates/{id}/interviews/{iid}/transcript.md  ← 复盘对话/生成评价时
  candidates/{id}/interviews/{iid}/eval_report.md ← 查阅历史评价时
  candidates/{id}/interviews/{iid}/session.json   ← 需要元数据/录音路径时
```

### 历史数据注入策略

不同场景下，Agent 注入上下文的方式不同：

| 场景 | 注入内容 |
|---|---|
| 实时面试（Interview Agent 系统提示词） | `profile.md` + 当次 `questions.md`，**不注入**历史 transcript |
| 面试官自然语言询问候选人情况 | 先读 `interviews/index.md` 摘要；根据关键结论判断是否深读历史 `eval_report.md` |
| 生成评价报告 | 注入当次 `transcript.md`（+ 可选：历史 `eval_report.md` 摘要作为参照） |
| 为下一轮准备问题 | 读历史 `eval_report.md` 中的优势/不足字段，调整新的问题方向 |

实时面试阶段严禁注入历史 transcript，原因：历史对话与本轮候选人回答混淆会干扰追问建议的准确性，且历史 transcript 通常较长，token 成本显著。

## CRUD 操作边界

本方案的增删改查由两个执行主体分工：

| 操作 | 执行主体 | 适用实体 |
|---|---|---|
| 新建候选人 | REST API → `MemoryModule` | 上传简历触发，结果确定 |
| 删除候选人 | REST API → `MemoryModule` | 删除整个 `candidates/{id}/` 目录 |
| 创建/结束面试会话 | REST API → `MemoryModule` | 开始时调 `start_interview` 写 `session.json` + `questions.md`；结束时调 `finish_interview` 写 `transcript.md` 并更新各 index |
| 修改问题列表 | Agent（通过文件写工具） | 语义理解后写回 `questions.md` |
| 生成/更新评价报告 | Agent（EvalAgent） | 写 `eval_report.md`，更新 `interviews/index.md` |
| 查询候选人列表 | REST API（解析 `candidates/index.md` YAML frontmatter → JSON） | 前端列表页直接消费 JSON；`MemoryModule.search_candidates` 负责解析 |
| 查询历史面试详情 | REST API（解析 frontmatter）或 Agent（自然语言问答） | 取决于是结构化展示还是叙事问答 |

Agent 写文件时，工具层（而非 Agent 直接输出原始 Markdown）负责组装最终格式：工具接受结构化参数，内部套用固定模板写入。这是防止格式损坏的主要手段，无需引入独立 schema 校验库。

## 运行时写入策略

面试过程中，`InterviewSession` 对象保持全程驻内存（现有行为不变）。写入时机：

| 事件 | 写入内容 |
|---|---|
| 简历上传解析完成 | `profile.md`，更新 `candidates/index.md` |
| 面试开始（`start_interview`） | `session.json`（stage=interviewing），`questions.md` |
| 面试结束（`finish_interview`） | `transcript.md`，更新 `session.json`（stage=completed, end_time, context_summary），更新 `interviews/index.md`，更新 `candidates/index.md` 中对应候选人的 `latest_interview` |
| 评价生成完成 | `eval_report.md`，更新 `interviews/index.md` |
| 候选人删除 | 删除整个 `candidates/{id}/` 目录，更新 `candidates/index.md` |

所有文件写入使用 **原子写入**（`mkstemp` + `os.replace`），与现有 `UserMemoryStore` 保持一致。

- `src/web/` 层（除上述局部改动外），包括：
  - `GET /api/recordings/{session_id}/rounds/{round_number}`：录音文件按约定路径 `recordings/{session_id}/rounds/round_{N:03d}_{source}.wav` 拼接，路由直接由参数构造路径，不查询存储层，不受重构影响
  - `DELETE /api/candidates/{candidate_id}` 路由中的"候选人正在使用中"检查：在内存 session 层判断（检查当前活跃会话的 `candidate_id`），与存储层无关，不受重构影响

## Web 界面数据展示策略

前端展示结构化数据时，采用以下混合方案：

| 数据类型 | 方案 | 说明 |
|---|---|---|
| 候选人列表 | Option A：解析 `candidates/index.md` → REST JSON | 确定性，无 LLM 成本 |
| 候选人简历详情 | 直接返回 `profile.md` 正文 Markdown，前端 `ui.markdown()` 渲染 | 与评价报告正文策略一致；`CandidateProfile` 不再含嵌套结构字段 |
| 问题清单 | Option A：解析 `questions.md` frontmatter（YAML 结构化列表） | 结构固定，直接反序列化 |
| 面试进度/状态 | Option A：读 `session.json` | 已是 JSON，直接返回 |
| 评价报告元字段（评分、结论） | Option A：读 `eval_report.md` frontmatter | 结构化字段在 frontmatter 中 |
| 评价报告正文 | 直接返回 Markdown，前端渲染 | 叙事内容，无需 LLM 二次处理 |
| 自然语言问答（"这个候选人适合吗？"） | Option B：Agent 读文件后生成回答 | 语义理解，必须走 Agent |

**原则**：结构化展示走 Option A（`MemoryModule` 解析 frontmatter，FastAPI 序列化为 JSON）；叙事分析走 Option B（Agent）。不应要求 LLM 输出纯 JSON 用于 UI 列表渲染——成本高且不稳定。

## 潜在风险与对策

| 风险 | 对策 |
|---|---|
| `index.md` 与实际目录不同步 | `MemoryModule` 每次写入后同步更新；提供 `rebuild_index()` 内部工具方法（不对外暴露为 REST 接口），从目录结构重建 index：扫描所有 `profile.md` frontmatter 重新生成 `candidates/index.md`，扫描所有 `session.json` 重新生成 `interviews/index.md`。可在启动时可选调用，也可通过管理脚本手动触发 |
| 多文件操作的部分失败（`finish_interview` 需写 3 个文件） | 按固定顺序写入：先写数据文件（`transcript.md` → `session.json`），最后更新 index（`interviews/index.md`）。index 滞后写入，崩溃时 index 缺条目但数据文件完整；`rebuild_index()` 可自动修复。不引入跨文件事务 |
| 面试进行中崩溃导致 transcript 数据丢失 | transcript 仅在面试结束时一次性写入（运行时驻内存），崩溃会丢失当次所有 rounds。对单用户本地工具可接受（面试本身也需重新进行）。如后续需要保护，可每 round 结束时追加写入临时文件，结束时合并——本次暂不实现 |
| 写入途中崩溃导致单文件损坏 | 所有文件写入使用原子写入（`mkstemp` + `os.replace`），崩溃只影响临时文件，目标文件始终完整 |
| 并发写入（极低概率，单用户工具） | 沿用 `UserMemoryStore` 的原子写入模式，不引入额外锁 |
| 候选人搜索性能 | 搜索时读 `candidates/index.md` frontmatter（O(1)），in-memory 关键字过滤；如需全字段搜索，枚举 `candidates/*/profile.md` frontmatter，百量级下可接受，无需额外索引 |
| Agent 写入格式错误 | 工具层负责组装 Markdown，Agent 只传结构化参数；`MemoryModule` 写入前校验必填 frontmatter 字段 |

## 验收标准

以下标准可量化、可自动化验证：

1. **候选人 CRUD**：`save_candidate` → `get_candidate` 往返后字段一致；`delete_candidate` 后目录不存在且 `candidates/index.md` frontmatter 中无该 ID
2. **index 一致性**：`save_candidate` 写入后，`candidates/index.md` frontmatter 的 `candidates` 列表长度 +1，且包含正确的 `id` 和 `name`
3. **面试生命周期**：`start_interview` → `session.json` 存在且 `stage=="interviewing"`；`finish_interview` → `transcript.md` 存在，`session.json` 中 `stage=="completed"`，`interviews/index.md` frontmatter 中有对应条目，`candidates/index.md` 中对应候选人的 `latest_interview` 已更新
4. **问题清单往返**：写入 `questions.md` 后，`get_latest_question_plan` 返回的列表与原始 `list[InterviewQuestion]` 在 `id`/`question`/`is_covered` 字段上完全一致
5. **评价报告**：`save_eval_report` 后，`get_eval_report` 返回的 `overall_score` 和 `recommendation` 与写入值一致；`interviews/index.md` 中对应行的评分和关键结论已更新
6. **index 重建**：手动删除 `candidates/index.md` 后调用 `rebuild_index()`，重建后 `get_candidate_by_name` 和 `search_candidates` 仍能正确返回结果
7. **崩溃兼容**：`transcript.md` 存在但 `interviews/index.md` 缺对应条目时，`rebuild_index()` 可修复，`get_candidate_history` 能返回该条面试摘要
