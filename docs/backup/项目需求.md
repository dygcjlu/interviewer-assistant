# 面试助手 - 项目需求文档

## 一、系统定位

- 面向**技术面试**场景的面试官辅助工具
- **辅助模式**：面试官全程主导提问，系统在旁实时提供追问建议、记录和分析，系统不直接与候选人对话
- 个人使用，不需要多用户/账号体系
- 通过 Web 页面进行交互
- **纯本地运行**：在面试官自己的电脑上启动服务，浏览器访问本地地址
- **串行面试**：一次只进行一场面试，不需要并发支持
- Python 后端，多 Agent 架构（自建框架，不依赖 LangGraph/AutoGen 等）
- 国产 LLM 为主（通义千问/DeepSeek/文心等），抽象接口层支持灵活切换
- SQLite 持久化存储
- STT 抽象接口层，当前实现为百度实时语音识别，可替换

## 二、核心功能模块

### 1. 简历解析与题目提炼

- 支持上传 PDF 简历，兼容文本类 PDF 和截图类 PDF（OCR）
- 解析出候选人的关键信息：技术栈、项目经历、工作年限、教育背景等
- 生成候选人画像摘要
- 根据简历内容自动生成针对性面试题目清单
- 题目按维度分类：基础知识、项目深挖、系统设计、编码能力等

### 2. 实时面试辅助

- **MVP 仅支持远程面试场景**（腾讯会议/Zoom 等）
- 通过系统 Loopback 采集会议音频（候选人发言），麦克风采集面试官发言，双声道物理分离实现说话人区分
- 实时语音转文字（STT）
- 根据候选人的实时回答，生成追问建议供面试官参考
- **追问建议流式显示**：LLM 生成的追问问题以打字机效果逐字（token）流式呈现在 Web 页面侧栏，面试官无需等待完整生成即可开始阅读，降低感知延迟
- 面试官可以采纳建议，也可以忽略
- Web 页面侧栏实时显示转写内容和追问建议

### 3. 面试评价

- 面试结束后生成结构化评价报告
- 分维度评分：基础知识、项目经验、系统设计、编码能力、沟通表达等
- 附带文字叙述：优势、不足、总体建议（录用/待定/不录用）
- 引用面试中的具体对话作为评价依据

### 4. 记忆模块

记忆分为**短期记忆**和**长期记忆**两层：

#### 短期记忆（in-session）

存在运行时的 `InterviewSession` 对象中，随面试进行实时更新，面试结束后归档到数据库：

- `covered_dimensions`：已覆盖的考察维度集合，避免重复提问
- `context_summary`：早期对话的压缩摘要，供上下文管理器维护
- 当前轮次的工作笔记（候选人在本场面试中暴露的短板/亮点）

#### 长期记忆（cross-session）

持久化到 SQLite，支持跨面试会话的数据积累：

- **候选人历史**：每次面试的对话记录、LLM 摘要、评价报告、录音路径
- **候选人档案**：从历次面试中提炼出的技术短板、优势特征等关键标签，面试准备阶段注入固定区
- 支持按候选人姓名/时间检索历史面试记录

#### 再次面试时的历史注入

当面试官再次面试同一位候选人时，系统在**面试准备阶段**从长期记忆中读取该候选人的历史档案（上次面试评价、已知短板等），生成历史摘要，作为候选人信息的一部分写入 prompt 固定区，全程占用固定 token 预算。

#### 面试后记忆整合

评价报告生成完毕后，系统后台异步执行记忆整合任务：将本次面试识别到的关键信息（新暴露的技术短板、突出亮点、考察维度覆盖情况）追加写入候选人长期档案，为下次面试提供更丰富的历史背景。

- ~~**题库积累**：记录题目的使用情况和效果，逐步优化出题策略~~ → 移至 Phase 2

### 5. 上下文管理模块

面试持续 30~60 分钟，对话轮次多，需要专门的上下文管理机制来控制 LLM 输入质量和 token 开销。

#### 数据模型

每一轮对话由以下结构组成：

- **面试官问题**（STT 转写 / 手动输入）
- **候选人回答**（STT 转写）
- **LLM 建议**（系统生成的追问建议）

每轮作为一条 user 消息发送给 LLM，LLM 返回下一步追问建议。

#### 上下文窗口策略

- **滑动窗口**：保留最近 N 轮完整对话（如最近 5~8 轮），确保 LLM 掌握当前话题的连续性
- **早期对话摘要**：超出窗口的早期对话由 LLM 自动压缩为摘要，保留关键信息（候选人暴露的技术短板、已覆盖的考察维度等）
- **固定上下文区**：候选人简历摘要、面试题目清单、当前考察维度始终置于上下文头部，不被截断

#### 三阶段压缩流程

触发摘要压缩时，分三个阶段依次执行，直至上下文满足 token 预算：

1. **Phase 1 — 剪枝**：去除重复或低价值内容（如中间态的 LLM 建议文本）
2. **Phase 2 — head/tail 截断**：保留最早 K 轮（提供面试背景）和最近 M 轮（保证连续性），丢弃中间轮次
3. **Phase 3 — LLM 摘要**：将待压缩的轮次发给 LLM，生成结构化摘要，写入摘要区

**SUMMARY_PREFIX**：摘要区内容开头固定添加标头（如 `[以下为早期对话的压缩摘要，非原始记录]`），使 LLM 明确区分摘要与原始对话，避免对摘要中的内容进行不必要的细节追问。

**压缩可行性检查**：执行 LLM 摘要前，检查所选模型的上下文窗口是否足以容纳本次压缩请求。若模型窗口较小（如 32K 配置），自动提前降低压缩触发阈值，避免压缩操作本身超出窗口限制。

**压缩后重注入**：压缩完成后，确保固定区（候选人简历摘要、题目清单、已覆盖维度）在新的上下文中完整重注入，不被截断。

#### token 预算控制

- 根据所选模型的上下文窗口动态调整单次请求的 token 预算（国产模型常见窗口：DeepSeek 128K、通义 32K~128K），超出预算时优先压缩早期摘要
- 实时追踪累计 token 消耗，前端可展示

#### 上下文感知能力

- 追踪已提问的题目和已覆盖的维度，避免重复提问
- 根据候选人回答深度决定是继续深挖还是切换话题
- 在上下文摘要中标注候选人的关键表现（亮点/疑点），供评价 Agent 最终使用

## 三、多 Agent 协作设计

系统采用多 Agent 架构，通过 Orchestrator 管理 Agent 切换。**Agent 之间不是固定线性流转，而是支持按指令自由切换**，由面试官通过前端操作驱动。

### Agent 列表

| Agent | 职责 | 典型使用阶段 |
|-------|------|------------|
| 简历分析 Agent | 解析简历，提炼候选人画像，生成面试题目清单 | 面试准备阶段 |
| 实时面试 Agent | 接收实时转写文本，分析候选人回答，生成追问建议 | 面试进行阶段 |
| 评价 Agent | 汇总面试全过程对话，生成结构化评价报告 | 面试总结阶段 |

### Agent 切换方式

- **自由切换**：面试官可通过前端指令在任意 Agent 之间切换（需满足前置条件）
- **典型流程**：简历分析 → 实时面试 → 评价（最常见路径，但不是唯一路径）
- **灵活场景**：面试中途可切回简历分析查看信息，评价阶段可切回面试追问等

### Agent 共享数据

- 所有 Agent 通过共享的 `InterviewSession` 对象读写数据
- 候选人简历信息、面试对话记录、记忆模块等均在 Session 中统一管理
- Agent 之间不直接通信

## 四、Agent 框架核心能力

系统需具备通用 Agent 框架的核心功能：

- **Skill 模块**：基于文件系统的可扩展能力单元。每个 Skill 是一个独立目录（`skills/{skill_name}/SKILL.md` + 可选模板），运行时动态扫描加载。system prompt 中只注入 Skill 索引（名称 + 一句话描述），Agent 需要时再加载完整内容，避免 prompt 过长。

  内置面试场景 Skill：
  - `deep_dive`：技术细节深挖追问技巧（如何判断回答深度、如何引导候选人展开）
  - `dimension_switch`：考察维度切换时机判断（何时结束当前维度、如何自然过渡）
  - `behavioral_probe`：行为面试追问技巧（STAR 结构验证、真实性判断）
  - `resume_anchor`：以简历具体项目/经历为锚点展开提问（绑定候选人亲身经历，降低套答概率）

- **工具模块**：Agent 可调用的外部工具（如简历解析、语音转写等），通过 `ToolRegistry` 统一注册和调度
- **记忆模块**：短期记忆（in-session）+ 长期记忆（SQLite 持久化），支持跨面试会话的数据积累
- **上下文管理模块**：固定区 + 摘要区 + 滑动窗口三区结构，三阶段摘要压缩，控制长面试场景下的 token 开销和上下文质量
- **Prompt 构建器（PromptBuilder）**：按固定层次顺序构建 system prompt，session 级缓存，压缩后重建。构建层次：Agent 身份定义 → Skill 索引 → 工具指引 → 候选人长期记忆 → 面试固定区（简历摘要、题目清单、已覆盖维度）
- **实时输出**：支持流式输出，面试辅助信息实时推送到前端
- **可观测性**：结构化轮转日志（`logs/app.log` / `logs/app.error.log`）；每场面试生成轨迹文件（`trajectories/{session_id}.jsonl`），每行记录一轮的输入/输出/token 用量，便于调试和复盘

## 五、典型使用流程

1. 面试官上传候选人简历 PDF
2. 系统解析简历，展示候选人画像，生成推荐题目清单
3. 面试官审阅/调整题目，确认后进入面试模式
4. 面试开始：系统监听远程会议音频（Loopback + 麦克风双声道），实时转写，自动区分发言人
5. 系统根据候选人回答，在 Web 页面侧栏实时显示追问建议
6. 面试官自行决定是否采纳建议进行追问
7. 面试结束：系统生成评价报告（评分 + 文字总结）
8. 报告存档，候选人信息记入记忆模块

## 六、待讨论的扩展功能

以下功能暂不纳入首期范围，后续根据实际使用情况决定是否加入：

- **面对面面试场景**：本地音频采集，支持候选人和面试官面对面沟通，需配合说话人分离或声纹识别实现发言人区分
- **声纹识别**：自动区分面试官和候选人的发言，替代双声道物理分离方案
- **题库积累**：记录题目的使用情况和效果，逐步优化出题策略
- **岗位信息输入**：面试官提前录入岗位要求（JD），使出题和评价更有针对性
- **题目编辑能力**：系统生成题目后，面试官可增删改题目，或从题库中手动挑选
- **面试中的实时标注**：面试官在面试过程中对候选人某个回答打标（如"回答很好"/"答偏了"），辅助最终评价
- **面试模板**：预设不同的面试模板（如 Java 后端、前端、算法等），加速准备流程
- **数据导出**：评价报告导出为 PDF/Word

## 七、技术约束与选型

| 维度 | 选型 | 说明 |
|------|------|------|
| 部署环境 | 纯本地运行，Windows | 面试官电脑上启动服务，浏览器访问 localhost |
| 后端 | Python 3.12+，asyncio | 单进程异步架构 |
| 前端 | 待定（Vue / React 均可） | 轻量方案，核心是实时展示转写和建议 |
| LLM | 国产模型为主 | 通义千问/DeepSeek/文心等，抽象接口层支持切换 |
| STT | 百度实时语音识别（已实现） | 预留抽象接口，后续可替换为 FunASR/Whisper 等 |
| 持久化 | SQLite | 单文件数据库，本地场景足够 |
| 实时通信 | WebSocket | 后端向前端推送转写文本和追问建议 |
| Agent 框架 | 自建 | 不依赖 LangGraph/AutoGen 等第三方框架，完全可控 |
| 配置管理 | 配置文件（`config.yaml`）+ `.env` 文件双层 | 基础配置在配置文件中，敏感信息（API Key）在 `.env` 中，禁止硬编码 |

## 八、非功能性需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 追问建议首 token 延迟 | 候选人说完后 **5 秒内** | 包含 STT 最终结果返回 + LLM 流式生成首 token 的时间 |
| 追问建议流式体验 | 首 token 到达后**逐 token 实时推送到前端** | LLM 流式输出 → WebSocket 逐片段下发 → 前端打字机效果渲染，全链路流式，无需等待完整生成 |
| STT 准确率 | 依赖百度实时 ASR 原生准确率 | MVP 不做额外后处理或纠错 |
| 数据存储 | 对话记录持久化到 SQLite，完整录音 + 按轮次切片存储到本地文件系统 | 数据库中仅存录音文件路径引用，不存音频二进制 |
| 配置管理 | 配置文件 + `.env` 双层 | 基础配置在 `config.yaml`，API Key 等敏感信息在 `.env`，禁止硬编码 |
| 单场面试容量 | 支持 30~60 分钟连续面试 | 上下文管理确保长面试不会 OOM 或 token 溢出 |

## 九、降级与容错策略

| 异常场景 | 降级策略 |
|----------|----------|
| STT 连接断开 | 自动重连（最多 3 次），重连失败后前端提示"语音识别中断"，面试官可切换为手动输入文字继续面试 |
| LLM API 超时/限流 | 前端显示"建议生成中..."，超时 10 秒后跳过本轮建议，不阻塞面试流程 |
| 音频设备不可用 | 启动时检测可用设备，无可用设备则提示并降级为纯手动输入模式（仅文字交互） |
| LLM 返回异常内容 | 丢弃本轮建议，前端不展示，记录日志供排查 |
| 简历解析失败 | 提示解析失败原因，面试官可手动输入候选人关键信息后继续 |
| 录音写入失败 | 记录日志告警，不影响面试主流程继续 |

## 十、MVP 与 Phase 2 划分

### MVP（Phase 1）

- 简历解析（PDF 文本提取 + OCR）与候选人画像生成
- 自动生成针对性面试题目清单
- 远程面试音频采集（系统 Loopback + 麦克风双声道分离）
- 实时 STT 转写（百度实时语音识别）
- 实时追问建议生成（国产 LLM）
- 上下文管理（滑动窗口 + 摘要压缩 + 固定上下文区）
- 面试结束后生成结构化评价报告
- **完整录音 + 按轮次切片**：保存完整面试录音和按轮次切片的音频片段，支持面试后回放核验
- 候选人历史记录 + 面试记录检索（SQLite）
- Web 页面：面试控制台 + 侧栏实时转写/建议展示 + 评价报告页录音回放
- 降级与容错：STT 断连重连、LLM 超时跳过、手动输入兜底、录音写入失败不阻塞主流程

### Phase 2（按优先级排列）

- 面对面面试场景 + 说话人分离/声纹识别
- 题库积累与题目效果记录
- 岗位 JD 输入，针对性出题和评价
- 题目编辑与手动挑选
- 面试中实时标注
- 面试模板（Java 后端、前端、算法等）
- 数据导出 PDF/Word
