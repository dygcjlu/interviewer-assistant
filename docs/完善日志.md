# 任务：完善面试助手日志体系

## 角色与目标

你是本仓库的**后端日志完善**开发 Agent。目标：在不改变业务行为的前提下，让运维与排障人员能**仅凭日志**还原一次请求的完整路径（入口 → 关键步骤 → 结果/异常），并统一日志落盘方式，避免错误与上下文割裂。

**成功标准（全部满足才算完成）：**

1. 任选一条核心链路（如 `POST /api/resume/upload` 或 WebSocket `manual_input`），在 `logs/app.log`（或下文约定的主日志文件）中能看到：**请求标识、session_id、阶段起止、成功/失败及耗时**。
2. `ERROR` 及以上级别**同时**出现在主日志与错误专用日志中，且主日志中错误前后有 INFO 上下文（非孤立一行）。
3. 无新增 `print()`；敏感信息（API Key、完整简历正文、录音二进制）不出现在日志中。
4. 现有测试通过；若改动 logging 初始化，补充或更新 `tests/` 中相关用例。

---

## 项目背景（只读，勿改业务语义）

- **栈**：Python 3.12+ / FastAPI / asyncio 单进程；日志使用标准库 `logging`。
- **入口**：`src/main.py` 中 `logging.basicConfig(level=INFO, format="%(asctime)s %(levelname)s %(name)s: %(message)s")`。
- **开发启动**：`scripts/start-dev.ps1` 将 stdout → `logs/backend.out.log`，stderr → `logs/backend.err.log`（当前 Python INFO 多走 stderr，导致「错误日志文件」里混有大量 INFO，主输出文件几乎为空）。
- **已有日志**：`Orchestrator` 会话创建/切换、`ResumeAgent` 激活等少量 INFO；`routes.py` 等业务路径几乎无结构化日志；`httpx`/`openai` 第三方库日志较吵。

**核心链路（优先覆盖，按优先级排序）：**

| 优先级 | 模块/路径 | 关键步骤示例 |
|--------|-----------|--------------|
| P0 | `src/web/routes.py` | 每个 REST：收到请求 → 委托 Orchestrator/Memory → HTTP 状态与耗时 |
| P0 | `src/web/websocket.py` | 连接/断开；入站 `type`；出站广播类型；异常 |
| P0 | `src/agents/orchestrator.py` | `create_session` / `switch_agent` / `handle_request` 起止与结果 |
| P1 | `src/agents/*_agent.py` | `handle_request` 按 `request.type` 记录开始/结束/失败原因（不含 LLM 全文） |
| P1 | `src/audio/transcription.py` | `finalize_round` / `flush_pending_round`；轮次写入 `session.rounds` |
| P1 | `src/llm/client.py` | 调用起止、模型名、token 用量（若有）、失败与重试（不记录 prompt 全文） |
| P2 | `src/storage/memory_module.py` | `save_interview`、候选人持久化等写库操作 |
| P2 | `src/audio/manager.py` | 音频 start/stop/pause 及失败（已有部分日志可补全） |

详细模块职责见根目录 `CLAUDE.md`，实现时遵守其中「日志用 `logging`、禁止 `print`」等规范。

## 需求 1：核心路径「可追踪」日志

### 1.1 统一关联字段

在 Web 层为每个 REST 请求生成 **`request_id`**（如 UUID 短码），通过 `contextvars.ContextVar` 或 FastAPI 中间件注入，使同一次请求内所有 `logger` 输出可 grep。

建议日志前缀或结构化字段（任选一种，全项目一致即可）：

```
session_id=<uuid> request_id=<id> agent=<name> op=<operation>
```

WebSocket 连接使用 **`connection_id`**；消息处理时附带 `session_id`（若已绑定）。

### 1.2 每个关键步骤必须记录的内容

| 时机 | 级别 | 必须包含 |
|------|------|----------|
| 步骤开始 | INFO | `op` 名称、输入摘要（参数名/类型/长度，非敏感值） |
| 步骤成功 | INFO | 结果摘要（如 `rounds_count`、`questions_count`、`status_code`） |
| 步骤失败 | ERROR | 异常类型 + 消息；业务异常用 `exc_info` 仅当需要栈时 |
| 外部 IO（LLM/DB/文件） | INFO | 起止 + **耗时 ms** |

### 1.3 禁止与限制

- 不记录：完整 PDF 文本、LLM prompt/response 全文、`.env` 密钥、Authorization 头。
- 简历/转写：最多记录文件名、字节大小、文本 **前 80 字符** 或 **字符数**。
- 不为此任务重构业务逻辑或引入新依赖（如 structlog），除非仅用于 logging 配置且改动极小。

### 1.4 第三方库噪音

在 logging 配置中合理设置 `httpx`、`httpcore`、`openai` 等 logger 的 level（建议 WARNING），避免淹没业务日志；**不要**完全关闭 uvicorn 访问日志，可降为 INFO 并保留。

## 需求 2：日志文件策略（主日志 + 错误副本）

### 2.1 目标行为

| 文件 | 内容 |
|------|------|
| `logs/app.log`（主日志） | DEBUG/INFO/WARNING/ERROR/CRITICAL **全部**写入（开发默认 INFO） |
| `logs/app.error.log`（错误副本） | 仅 ERROR 及以上（与主日志中错误行一致） |
| `logs/backend.out.log` / `backend.err.log` | 保留给 `start-dev.ps1` 进程级重定向；**应用级**日志以 `app.log` 为准 |

实现方式建议（二选一，优先 A）：

- **A.** 在 `src/main.py`（或新建 `src/logging_config.py`）用 `logging.handlers.RotatingFileHandler` 配置双文件 + 控制台；`basicConfig` 改为 `dictConfig` 或显式 `getLogger` 根配置。
- **B.** 仅改 `start-dev.ps1`：合并 stdout/stderr 到单一 `logs/dev.log`，并在应用内仍配置 `app.log` + `app.error.log`。

**硬性要求：** 任意 `logger.error(...)` / `logger.exception(...)` 必须出现在 **`app.log` 中错误发生前后的 INFO 上下文附近**（靠同一 `request_id` 关联，而非只在独立 error 文件里）。

### 2.2 轮转与目录

- 单文件上限建议 10MB，保留 5 个备份；`logs/` 已在 `.gitignore` 时勿提交日志文件。
- 启动时确保 `logs/` 目录存在（与 `start-dev.ps1` 行为一致）。
