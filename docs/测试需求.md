# 面试助手 — 端到端功能测试指令

## 背景

本项目已按照 `docs/项目需求.md` 和 `docs/arc/` 目录下的概要设计文档从零完成代码编写，尚未进行集成测试和端到端测试。

请按以下计划逐项执行**端到端功能测试**。测试方式：启动后端服务 + 打开前端 Web 界面，通过界面交互触发功能，结合日志和实际返回结果判断测试是否通过。这不是单元测试，而是完整的功能路径验证。

---

## 一、测试前置准备

### 1.1 环境信息

- 操作系统：Windows 10/11
- Python：3.12+
- LLM 提供商：通义千问（Qwen）
  - `QWEN_API_KEY=sk-f59f99e7d83d4abc958f8477efbe1392`
  - `QWEN_API_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1`
  - 推荐模型：`qwen-plus` 或 `qwen-turbo`（支持 Function Calling）
  - 官方文档：https://bailian.console.aliyun.com/cn-beijing?tab=api#/api/?type=model&url=3016807

### 1.2 项目结构关键路径

```
src/main.py         # 启动入口
config.yaml         # 基础业务配置
.env                # 敏感配置（API Key）
frontend/           # Vue 3 前端
logs/               # 运行日志（agent.log / error.log）
trajectories/       # 面试轨迹 JSONL
```

### 1.3 启动步骤

**第一步：检查并创建 `.env` 文件**

在项目根目录检查是否存在 `.env` 文件。若不存在，参考 `.env.example` 创建，内容至少包含：

```
QWEN_API_KEY=sk-f59f99e7d83d4abc958f8477efbe1392
QWEN_API_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
```

**第二步：检查 `config.yaml`**

确认 `config.yaml` 中 LLM 配置的 `model` 字段指向有效的 Qwen 模型（如 `qwen-plus`），以及其他必填配置项（`context.max_tokens`、`audio` 相关等）是否有合理默认值。

**第三步：安装依赖**

```powershell
cd d:\interview\interviewer-assistant
pip install -e .
# 或
pip install -r requirements.txt
```

**第四步：构建前端（如尚未构建）**

```powershell
cd frontend
npm install
npm run build
cd ..
```

**第五步：启动后端服务**

```powershell
cd d:\interview\interviewer-assistant
python -m src.main
# 或
uvicorn src.main:app --host 0.0.0.0 --port 8000
```

服务启动后，在浏览器打开 `http://localhost:8000`，确认前端页面正常加载。

**第六步：开启日志监控**

在另一个终端窗口实时追踪日志：

```powershell
Get-Content -Path "logs\agent.log" -Wait -Tail 50
```

## 二、测试计划（按功能模块）

### T-01：服务启动与前端加载

**目标**：确认服务正常启动，前端四个页面均可访问。

**操作步骤**：
1. 执行上述启动步骤，观察控制台输出，确认无异常报错。
2. 浏览器访问 `http://localhost:8000`，确认首页（候选人列表页）正常渲染。
3. 依次导航到准备页（Prepare）、控制台页（Console）、报告页（Report），确认路由正常。
4. 打开浏览器开发者工具 → Network，确认无明显 4xx/5xx 请求错误。

**成功标准**：
- 后端进程无 `ERROR` 级别启动日志。
- 前端四个页面均能正常渲染，无白屏、无 JS 报错。
- `GET /api/session/current` 返回 200（即使 session 为空）。

---

### T-02：简历上传与候选人画像生成（ResumeAgent）

**目标**：验证 PDF 简历上传 → LLM 解析 → 候选人画像 + 题目清单生成的完整链路。

**测试数据**：准备一份简单的技术候选人 PDF 简历（内含技术栈、项目经历等信息）。若无现成 PDF，可创建一个包含文字的简单 PDF 用于测试。

**操作步骤**：
1. 在前端准备页（Prepare）找到简历上传入口。
2. 上传 PDF 文件，点击"解析"或"上传"按钮。
3. 等待解析完成（观察页面 loading 状态）。
4. 确认页面展示候选人画像摘要（技术栈、项目经历、工作年限等）。
5. 确认页面展示生成的面试题目清单，按维度分类（基础知识、项目深挖、系统设计等）。

**日志验证**：
- `agent.log` 中应出现 `ResumeAgent` 相关的 INFO 日志，包含工具调用和 LLM 响应记录。
- 确认无 `ERROR` 日志（尤其是 LLM API 调用失败）。

**API 验证**（可用 `curl` 或浏览器开发者工具）：
- `GET /api/resume/profile` → 应返回包含候选人信息的 JSON。
- `GET /api/interview/questions` → 应返回题目清单数组。

**成功标准**：
- 简历解析成功，画像和题目清单正常展示。
- LLM 实际调用 Qwen API（日志中可见 token 使用量）。
- 无超时、无 500 错误。

---

### T-03：面试启动与 Agent 切换（Orchestrator）

**目标**：验证从准备阶段切换到面试阶段的 Agent 状态机流转。

**前置条件**：T-02 已完成，候选人画像和题目清单已生成。

**操作步骤**：
1. 在准备页审阅题目清单（可选：调整题目顺序或内容）。
2. 点击"开始面试"按钮，触发 `POST /api/interview/start`。
3. 观察页面跳转到面试控制台（Console 页）。
4. 检查控制台页顶部状态栏，确认当前 Agent 为"实时面试 Agent"（InterviewAgent）。

**日志验证**：
- `agent.log` 中出现 `Orchestrator` 切换到 `InterviewAgent` 的日志。
- `InterviewAgent.on_activate()` 被调用的日志。

**API 验证**：
- `GET /api/session/current` → 返回的 `stage` 字段应为 `interview`（或类似枚举值）。

**成功标准**：
- 页面正确跳转，Agent 切换成功，无报错。

---

### T-04：WebSocket 连接与实时推送

**目标**：验证 WebSocket 连接正常建立，服务端可向前端推送消息。

**操作步骤**：
1. 在面试控制台页，打开浏览器开发者工具 → Network → WS，确认 `/ws/interview` WebSocket 连接已建立（状态为 101 Switching Protocols）。
2. 在浏览器控制台执行手动触发建议（见 T-06），观察是否收到 WebSocket 消息。
3. 观察 WebSocket 消息帧中是否出现 `transcript`、`suggestion`、`status` 类型的消息。

**成功标准**：
- WebSocket 连接成功建立，无立即断开。
- 前端能正确接收并展示 WebSocket 下行消息。

---

### T-05：手动输入文字触发追问建议（核心链路，不依赖音频）

**目标**：在不启用音频的情况下，通过手动输入候选人回答文字，验证 InterviewAgent 追问建议生成的完整链路。

> 注意：音频采集（WASAPI + 百度 STT）在 MVP 测试阶段可通过 Mock 实现或手动输入绕过。若系统支持"手动输入转写文本"模式，优先使用该方式。

**操作步骤**：
1. 在面试控制台页找到手动输入区域（若有）。
2. 输入一段模拟候选人回答的文字，例如：
   > "我在上一家公司负责了电商平台的订单系统，主要使用 Java Spring Boot，遇到的最大挑战是高并发下的库存超卖问题，我们通过 Redis 分布式锁解决了这个问题。"
3. 提交文本（点击"发送"或类似按钮）。
4. 观察侧栏是否出现流式打字机效果的追问建议。

**API 验证**（备用方式，若前端入口不明显）：

```powershell
curl -X POST http://localhost:8000/api/interview/suggest `
  -H "Content-Type: application/json" `
  -d '{"text": "我用Redis分布式锁解决了库存超卖问题"}'
```

**日志验证**：
- `agent.log` 中出现 `InterviewAgent` 调用 LLM 的日志，含 token 用量。
- `trajectories/{session_id}.jsonl` 文件中出现本轮的输入/输出记录。

**成功标准**：
- 追问建议在 5 秒内开始流式展示（首 token 到达）。
- 建议内容与候选人回答相关，体现追问逻辑（如询问 Redis 分布式锁的具体实现细节）。
- 无 LLM 超时错误，无 WebSocket 断连。

---

### T-06：手动触发建议按钮（SuggestionTrigger 手动模式）

**目标**：验证面试官手动点击"获取建议"按钮触发追问建议的路径。

**操作步骤**：
1. 在面试控制台页，找到"手动获取建议"或"追问建议"按钮。
2. 点击该按钮，观察追问建议区域出现流式文字输出。
3. 多次点击，确认每次都能独立触发一次新建议，且不出现重复请求叠加问题。

**成功标准**：
- 每次点击均能触发新一轮 LLM 推理并流式展示结果。
- 前端建议区域内容在新建议生成时正确更新（不叠加旧内容）。

---

### T-07：上下文管理 — 多轮对话 token 追踪

**目标**：验证多轮对话后上下文管理模块正确累积对话历史，token 用量被追踪。

**操作步骤**：
1. 连续执行 T-05 操作 5~8 次，每次输入不同的候选人回答内容。
2. 观察前端是否展示 token 用量信息（若有 token 显示区域）。
3. 检查日志，确认每轮对话后 `ContextManager.add_round()` 被调用。

**日志验证**：
- `trajectories/{session_id}.jsonl` 中，每轮记录的 `token_usage` 字段值单调递增（或每轮有独立 token 统计）。
- 无 `token budget exceeded` 或类似上下文溢出的错误日志。

**成功标准**：
- 10 轮对话后服务仍正常响应，无内存溢出或 token 超限错误。
- 历史对话内容被正确纳入新一轮的 LLM 请求上下文（追问建议质量随对话深入而提升，不出现"无视历史"的情况）。

---

### T-08：面试结束与评价报告生成（EvalAgent）

**目标**：验证面试结束后 EvalAgent 生成结构化评价报告的完整链路。

**前置条件**：已完成至少 3~5 轮模拟对话（T-05/T-06）。

**操作步骤**：
1. 在面试控制台页点击"结束面试"按钮，触发 `POST /api/interview/stop`。
2. 等待 Agent 切换到 EvalAgent 并生成评价报告（可能需要 10~30 秒）。
3. 页面跳转到报告页（Report），确认展示：
   - 分维度评分（基础知识、项目经验、系统设计等）
   - 文字叙述（优势、不足、总体建议）
   - 引用具体对话内容作为评价依据

**API 验证**：

```powershell
curl http://localhost:8000/api/interview/eval
```

应返回包含 `scores`、`summary`、`recommendation` 等字段的 JSON。

**日志验证**：
- `agent.log` 中出现 `EvalAgent` 工作的日志。
- 确认评价报告被存入 SQLite（`storage/database.py` 相关的 INSERT 日志）。

**成功标准**：
- 评价报告成功生成，内容结构完整。
- 报告内容与本次面试对话内容相关（不是无关的通用评语）。
- 报告数据持久化到 SQLite，重启服务后仍可查询。

---

### T-09：候选人历史记录与长期记忆

**目标**：验证面试数据持久化和历史记录查询功能。

**前置条件**：T-08 已完成，至少完成一次完整面试流程。

**操作步骤**：
1. 在首页（Home）查看候选人列表，确认刚才完成面试的候选人出现在列表中。
2. 点击该候选人，进入历史记录页，确认面试记录可查询。
3. **重启服务**后，再次访问首页，确认候选人数据仍然存在（持久化验证）。

**API 验证**：

```powershell
# 查询候选人列表
curl http://localhost:8000/api/candidates

# 查询指定候选人历史
curl http://localhost:8000/api/candidates/{candidate_id}/history
```

**成功标准**：
- 候选人数据在服务重启后依然存在（SQLite 持久化正常）。
- 历史记录 API 返回正确的面试信息，包含对话轮次数、评价结论等。

---

### T-10：降级与容错验证

**目标**：验证关键异常场景的降级处理不会导致服务崩溃。

#### T-10a：LLM API 超时降级

**操作步骤**：
1. 临时修改 `.env` 中的 `QWEN_API_BASE_URL` 为无效地址（如 `http://invalid-host/v1`）。
2. 触发一次追问建议（T-05/T-06）。
3. 观察前端是否显示"建议生成中..."并最终跳过本轮建议（不阻塞面试流程）。
4. 恢复正确的 API 地址。

**成功标准**：
- 超时后（10 秒内）前端显示降级提示，面试主流程可继续。
- `error.log` 中记录超时异常，但无服务崩溃。

#### T-10b：无效简历文件处理

**操作步骤**：
1. 上传一个非 PDF 文件（如 `.txt` 或 `.jpg`）或内容为空的 PDF。
2. 观察前端是否显示友好的错误提示。

**成功标准**：
- 返回明确的错误提示，服务不崩溃，不影响后续操作。

## 五、Bug 修复原则

测试过程中发现的 Bug，修复前须遵循以下原则：

1. **以需求文档为基准**：优先参考 `docs/项目需求.md` 和 `docs/arc/` 目录下各概要设计文档，确认正确行为应该是什么，再动手修改代码。最终实现不得无故偏离已有设计文档的描述。

2. **设计文档优先于代码直觉**：若发现代码实现与设计文档不符，应以设计文档为准进行修复，而非根据"代码看起来更合理"的主观判断随意改动。

3. **如认为需求本身不合理**：若经过分析，认为需求文档或设计文档的某项要求存在明显问题（如技术上不可行、与其他约束冲突、用户体验较差等），允许采用更合理的方案，但**修复完成后必须向用户说明**：
   - 偏离了哪条需求或设计
   - 偏离的具体内容
   - 采用当前方案的理由

4. **修改范围最小化**：只修改导致 Bug 的最小代码路径，不趁机重构或"顺手优化"无关逻辑。每次修改后重新执行对应测试项验证修复效果。

## 六、备注

- **音频功能**：WASAPI 音频采集和百度 STT 依赖本机音频设备和百度 API，MVP 测试阶段优先通过手动输入文字（T-05）验证核心 AI 链路。若需测试音频全链路，须确保 `config.yaml` 中百度 STT 配置有效且本机有可用音频设备。
- **Qwen 模型选型**：建议使用 `qwen-plus`（平衡性能与成本）；若遇到 Function Calling 相关问题，可尝试 `qwen-max`。
- **前端 Dev 模式**：生产测试以 FastAPI 托管静态文件为准（`npm run build` 后）；若前端未构建，可临时用 `npm run dev`（端口 5173）配合后端 CORS 进行开发验证。
