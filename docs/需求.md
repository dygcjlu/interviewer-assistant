## 任务：代码审查与问题分析

### 背景

当前项目为面向单个面试官的本地面试辅助工具，支持管理多名候选人。技术栈：Python 3.12 + FastAPI + NiceGUI + SQLite + OpenAI 兼容 LLM。

### 你的目标

1. **深入 review 项目代码**，包括 `src/` 各模块、API 设计、数据流、前端交互逻辑。
2. **执行端到端测试**，覆盖主要用户流程（简历上传、面试准备、实时转写、追问建议、面试评价）。
3. **找出所有问题**，不限于下方列举的已知问题。主动发现隐患：性能瓶颈、数据一致性、错误处理缺失、UX 问题、安全隐患等。
4. **给出解决方案**，并将完整的问题清单与方案输出到 `docs/` 目录下的独立文档。

**再次强调**： 你不能局限于下方列出的已知问题，你应该进行全面且深入的分析，找出尽可能多的问题。
---

### 已知问题（必须解决）

#### 问题 1：候选人上下文未注入提示词

**现象**：在 Web 界面选择候选人后，向 Agent 提问候选人相关内容，Agent 回复称没有候选人信息。

**期望行为**：
- 选中候选人后，将候选人的**基本结构化信息**（姓名、学历、工作年限、技能标签等）自动注入 LLM system prompt 上下文。
- 完整简历内容不全量注入，改为让 Agent 通过工具调用（如读取本地 Markdown 文件）按需获取详细信息。(需要给agent提供工具，并在提示词中说明简历保存的路径)

---

#### 问题 2：简历导入无去重逻辑

**现象**：每次导入都生成新 candidate_id，同一候选人重复导入会重复解析 PDF。

**期望行为**：
- 自动提取 PDF 文件名作为候选人姓名标识（key）。(界面中提醒用户需要将名字作为简历文件名)
- 若同名候选人已存在，弹窗询问用户：**是否覆盖现有数据？**
  - 确认 → 重新解析 PDF，替换原有记录。
  - 取消 → 中止导入。

---

#### 问题 3：缺少候选人删除功能，左侧列表 UX 需改进

**现象**：只有新增候选人接口，无删除接口；候选人选择方式不直观。

**期望行为**：
- 在左侧面板展示候选人列表（支持至少 10 条，可滚动）。
- 列表支持**选中**（切换当前活跃候选人）和**删除**操作。
- 删除时同步清理：数据库记录 + 本地简历文件（PDF + Markdown）+ 所有关联数据（面试记录、评价报告等）。
- 设计候选人全生命周期管理机制，统一管理其所有关联资源。

---

#### 问题 4：缺少候选人详情查看界面

**现象**：选中候选人后无法查看其详细信息。

**期望行为**：
- 选中候选人后，可在主区域查看该候选人的所有信息：
  - 简历（Markdown 渲染）
  - 已生成的面试问题列表
  - 面试评价报告（若已生成）
  - 历史面试记录（若有）
- 候选人管理模块需覆盖其完整生命周期的所有信息聚合展示。

---

#### 问题 5：候选人基本信息未结构化提取

**现象**：PDF 解析后仅存储原始 Markdown 文本，基本信息散落在文本中，不便于程序处理和界面展示。

**期望行为**：
- PDF → Markdown 解析完成后，调用 LLM 对 Markdown 进行结构化提取，生成候选人基本信息对象，包含字段如：
  - `name`（姓名）
  - `education`（学历）
  - `years_of_experience`（工作年限）
  - `skills`（技能标签列表）
  - `current_position`（当前/最近职位）
  - `summary`（一句话简介）
- 将结构化数据持久化到数据库，供界面展示和提示词注入使用。

---

### 输出要求

分析完成后，在 `docs/` 目录下输出一份 `analysis.md`，内容包括：

1. **问题清单**：包含上述已知问题 + 你新发现的问题，每条说明：现象、根因、影响范围。
2. **解决方案**：每条问题对应的具体实施方案（涉及哪些文件、改什么逻辑）。
3. **优先级建议**：按影响程度排序，标注 P0/P1/P2。
4. **实施顺序建议**：考虑模块依赖关系，给出推荐的实施顺序。
