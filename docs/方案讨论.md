## 任务背景

我正在为一个**单面试官本地面试辅助工具**重构数据存储机制。重构方案的初稿见 `docs/数据存储方案重构.md`，该方案的核心思路是：**彻底移除 SQLite，改用纯文件（Markdown + JSON）存储所有候选人和面试数据**，由 Agent 通过文件层级导航按需读取。

核心业务流程如下：
1. 上传候选人 PDF 简历 → 解析为 Markdown → 提炼初始面试问题
2. 面试官可通过对话要求 Agent 新增或调整问题（Agent 写回文件）
3. 面试过程中，系统提示词中注入候选人简历与问题列表，Interview Agent 实时辅助
4. 面试支持多轮，每轮独立记录
5. 面试结束后，面试官可自然语言询问候选人情况，Agent 从文件读取数据回答
6. 可按需生成面试评价报告

## 需要讨论的开放问题

请针对以下问题逐条分析，给出你的判断和建议：

### 1. 完全移除 SQL 的可行性风险

初稿已决定移除 SQLite。请从以下角度评估这个决策：
- 本工具是单用户本地工具，移除 SQL 的最大实际风险是什么？
- 纯文件方案在哪些场景下会明显劣于数据库（例如面试数量增长到多少时性能开始下降）？
- 是否有必要保留任何形式的轻量索引（如 SQLite in-memory 或 JSON index 文件）作为查询加速层？

### 2. 纯文本方案下的 CRUD 设计

初稿中的写入时机已有说明，但增删改查的边界还需厘清：
- **增/删**：通过 REST API 直接操作文件（不经过 Agent），这对哪些实体适用？哪些不适用？
- **改**：Agent 通过读写文件工具修改（如更新问题列表），如何确保格式合规、不引入损坏？是否需要 schema 校验？
- **查**：Agent 查询时的导航路径是否足够高效？对于"列出所有候选人"或"找到某候选人最近一次面试的评价"这类查询，现有的两级 index 设计是否够用？

### 3. 多轮面试的数据模型

每位候选人可能经历多轮面试，当前方案用 `interviews/{id}/` 子目录区分。请分析：
- 如何在 interview agent 的系统提示词中高效注入"本次面试"数据，同时不带入过多历史轮次噪音？
- 历史面试数据何时对 agent 有用（应读入上下文），何时应忽略？
- `interviews/index.md` 的摘要粒度是否合适？是否需要调整字段？

### 4. Web 界面的数据展示重构

如果前端要展示结构化数据（候选人列表、问题列表、面试进度等），现有方案有两个选项：
- **选项 A**：后端直接解析 Markdown frontmatter 提供 REST API 返回 JSON
- **选项 B**：要求 LLM 输出结构化 JSON，前端消费

请分析两种方案各自适合哪些场景，以及是否可以混合使用。
