## 需求：丰富项目日志

通过日志追踪每个关键执行步骤，方便排查问题和还原执行路径。

---

### 一、总体原则

| 原则 | 说明 |
|------|------|
| 关键业务节点 | INFO 级别 |
| 异常 / 告警 | WARNING+ 级别（现有规范不变） |
| 数据截断 | 所有长文本字段统一截断为 **1000 字**，超出部分标注 `...(truncated)` |
| 来源标注 | 工具调用日志须通过 `[agent=xxx]` 上下文区分是哪个 Agent 发出的（复用现有 `bind_agent`） |

---

### 二、需要新增日志的关键节点

#### 2.1 工具调用（ToolRegistry.dispatch）

统一在 `ToolRegistry.dispatch` 的入口和出口各打一条 INFO 日志，覆盖所有 Agent 的工具调用。

| 时机 | 内容 |
|------|------|
| 调用前 | `tool_call: {tool_name}` + 截断后的 arguments JSON |
| 调用后 | `tool_result: {tool_name}` + 截断后的 result 字符串 + 耗时 ms |
| 异常时 | 已有 `logger.exception`，保持不变 |

> **区分来源**：`dispatch` 执行时，`bind_agent` contextvars 已经由各 Agent 设置好，日志格式里的 `[agent=xxx]` 字段自动体现调用来源，无需额外改动。

#### 2.2 MainAgent → ResumeAgent 分发

在 `delegate_to_resume_agent` 工具函数里补充日志：

| 时机 | 内容 |
|------|------|
| 调用前 | `delegating to resume_agent` + `pdf_path` + 截断后的 `instructions` |
| 返回后 | `resume_agent result` + 截断后的返回 JSON（profile 摘要、questions 数量） |

#### 2.3 LLM 调用（LLM 客户端）

在 LLM 客户端的 `chat` 方法入口和出口各打一条 INFO：

| 时机 | 内容 |
|------|------|
| 发送前 | `llm_request` + model 名 + 截断后的完整 messages JSON |
| 收到后 | `llm_response` + 截断后的完整 response content + 耗时 ms |

> 现有的 `llm_chat start` / `llm_chat done` 可升级为承载上述内容，无需新增日志行。

#### 2.4 追问建议生成结果（InterviewAgent）

在 `generate_suggestion` 完成后打一条 INFO：

| 内容 |
|------|
| `suggestion_generated` + 截断后的建议文本 + 耗时 ms |

#### 2.5 评价报告生成结果（EvalAgent）

在 `_generate_eval` 关键步骤补充：

| 时机 | 内容 |
|------|------|
| 开始生成 | `eval_start` + 参与评价的轮次数量 |
| 生成完成 | `eval_done` + 截断后的评价摘要（首 1000 字） + 耗时 ms |

---

### 三、实时转写：仅精简，不扩展

高频路径（STT 帧上传、partial segment）保持现有 DEBUG 级别，**不新增** INFO 日志。

仅在以下关键节点保留/升级为 INFO：

| 节点 | 说明 |
|------|------|
| 音频采集开始 / 停止 | `AudioManager.start` / `stop`（已有 INFO，保持） |
| 轮次确认（round finalized） | `TranscriptionManager.finalize_round`：`round_finalized` + 候选人文本前 100 字 + 面试官文本前 100 字 |
| 60s 静默强制结束 | 已有 WARNING，保持 |
| STT 连接错误 / 重连 | 已有 ERROR/WARNING，保持 |

---

### 四、不在本次范围内

- 日志格式、输出路径、轮转策略：现有配置不变
- `conversations/*.jsonl` JSONL 审计日志：现有机制不变
- 新增日志配置项（如 `LOG_LEVEL` 环境变量）：不在本次范围
